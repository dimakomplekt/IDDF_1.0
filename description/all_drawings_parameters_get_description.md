# Описание макроса

Макрос предназначен для пакетной синхронизации чертежей (.idw) с параметрами соответствующих моделей (деталей и сборок).

Обработка выполняется автоматически для всех подпапок проекта, за исключением архивных.

Макрос считывает пользовательские параметры моделей и переносит их в iProperties соответствующих чертежей.

___


## Общая логика работы

Макрос работает как пакетный обработчик файлов:

ROOT (папка проекта)  
↓  
Получаем список подпапок  
↓  
Исключаем архивную папку  
↓  
В каждой папке ищем *.idw  
↓  
Открываем каждый чертёж  
↓  
Синхронизируем с моделью  
↓  
Сохраняем  
↓  
Закрываем  
↓  
Показываем итог  


___


## Основные сведения

* Макрос запускается из любого файла внутри рабочей директории проекта.

* Обработка выполняется для всех подпапок текущей директории.

* Папка с именем `"1_Архив"` игнорируется.

* Чертежи открываются в фоновом режиме (без отображения UI).

* Обновление интерфейса Inventor отключается на время выполнения для ускорения работы.

* При повторном запуске макроса все данные актуализируются.


___


## Алгоритм работы макроса


### 1. Обход файловой структуры

1) Получаем путь текущего документа (корень обработки).

2) Получаем список всех подпапок.

3) Исключаем папку `"1_Архив"`.

4) В каждой папке ищем файлы формата `.idw`.

5) Если файлов нет — папка пропускается.



### 2. Обработка чертежей

Для каждого найденного чертежа:

1) Открываем файл в фоновом режиме.

2) Передаём документ в функцию `process_drawing`.

3) После обработки:
   - сохраняем файл
   - увеличиваем счётчик обработанных файлов

4) Закрываем документ без повторного сохранения.

5) В случае ошибки:
   - выводим сообщение с путём к файлу
   - продолжаем обработку остальных файлов



### 3. Синхронизация (process_drawing)

#### 3.1 Получение модели

1) Получаем активный лист чертежа.

2) Проверяем наличие видов:
   - если видов нет → выход

3) Берём первый (базовый) вид: ActiveSheet.DrawingViews(1)

4) Получаем связанную модель через: ReferencedDocumentDescriptor.ReferencedDocument


5) Если модель:
- не загружена
- ссылка сломана  

→ выполнение прекращается для текущего файла



#### 3.2 Чтение параметров модели

Все параметры читаются через безопасный геттер:

- `get_parameters()` — защита от отсутствия параметра
- `clean_string()` — очистка строки

Используются следующие параметры модели:

- "part_number" — код изделия  
- "part_name" — наименование  
- "part_type" — тип (деталь / сборка)  
- "part_developer" — разработчик  
- "developer_date" — дата разработки  
- "part_test" — проверяющий  
- "test_date" — дата проверки  
- "part_tech_control" — техконтроль  
- "tech_control_date" — дата техконтроля  
- "part_department_head" — начальник отдела  
- "department_head_date" — дата утверждения  
- "part_norms_control" — нормоконтроль  
- "norms_control_date" — дата нормоконтроля  
- "part_approved_by" — утверждающий  
- "part_approved_date" — дата утверждения  
- "part_company" — организация  



#### 3.3 Логика документа

Если: part_type = "Сборка"

то:

- к `part_number` добавляется суффикс `"СБ"` (если отсутствует)
- тип документа устанавливается как: "Сборочный чертеж"

#### 3.4 Запись в iProperties

Запись выполняется через безопасный сеттер: safe_properties_setter()


Особенности:

- защита от ошибок API
- нормализация значений
- отсутствие падений Inventor


##### Основные соответствия:

- Part Number ← part_number  
- Title ← part_name  
- Author ← part_developer  
- Creation Time ← developer_date  
- Checked By ← part_test  
- Date Checked ← test_date  
- Manager ← part_department_head  
- Authority ← part_department_head  
- Руководитель Дата ← department_head_date  
- Engr Approved By ← part_norms_control  
- Engr Date Approved ← norms_control_date  
- Mfg Approved By ← part_approved_by  
- Mfg Date Approved ← part_approved_date  
- Company ← part_company  

Дополнительно:

- Тип документа записывается только если определён



### 4. Завершение работы

После обработки всех файлов:

1) Включается обновление интерфейса.

2) Выводится сообщение: "Готово. Обновлено чертежей: <количество>"




___


## Особенности реализации

* Используется глобальное отключение `ScreenUpdating` для ускорения.

* Каждый файл обрабатывается в отдельном `Try/Catch`:
  - ошибки не останавливают выполнение

* Закрытие документов выполняется гарантированно (`Finally`).

* Макрос устойчив к:
  - отсутствию параметров
  - битым ссылкам моделей
  - ошибкам API Inventor


___


## Итог

Макрос реализует стабильный и безопасный pipeline:

→ файловая система  
→ чертёж  
→ модель  
→ параметры  
→ iProperties  

с полной защитой от сбоев и возможностью повторного запуска без побочных эффектов.
